### 🎯 Development Patterns & Best Practices

#### Server vs Client Components

**Server Components (default in app/ directory)**:
- Use for data fetching, static content, SEO
- Can directly access Supabase server client
- Cannot use hooks, event handlers, or browser APIs
- Import from `lib/supabase/server.ts`

**Client Components (marked with "use client")**:
- Required for interactivity, hooks, event handlers
- Use for forms, buttons, interactive UI
- Import from `lib/supabase/client.ts`
- Should be as small/specific as possible

#### Data Layer Architecture

**Server Actions** (`lib/actions/`):
- Handle all mutations (create, update, delete)
- Run on server with full security context
- Validate inputs with Zod schemas
- Return typed success/error responses
- Use `revalidatePath()` or `revalidateTag()` for cache invalidation

Example:
```typescript
'use server'
import { createClient } from '@/lib/supabase/server'
import { revalidatePath } from 'next/cache'

export async function createCabinet(data: CabinetInput) {
  const supabase = await createClient()
  const { data: cabinet, error } = await supabase
    .from('cabinets')
    .insert(data)
    .select()
    .single()
  
  if (error) return { error: error.message }
  revalidatePath('/admin/cabinets')
  return { data: cabinet }
}
```

**Data Fetchers** (`lib/data/`):
- Handle all queries (reads)
- Run on server in Server Components
- Return typed data or null
- Should be cached appropriately
- Use Supabase server client

Example:
```typescript
import { createClient } from '@/lib/supabase/server'

export async function getCabinets() {
  const supabase = await createClient()
  const { data, error } = await supabase
    .from('cabinets')
    .select('*')
    .order('name')
  
  if (error) return null
  return data
}
```

#### Authentication Patterns

- Supabase Auth with session-based authentication
- Use `@supabase/ssr` for server-side auth
- Auth state managed via middleware and server client
- Protected routes via layout.tsx checks
- OAuth callback handled in `/auth/callback/route.ts`

**Getting current user**:
```typescript
// In Server Component or Server Action
import { createClient } from '@/lib/supabase/server'

const supabase = await createClient()
const { data: { user } } = await supabase.auth.getUser()
```

#### Validation & Type Safety

- **All user inputs** must be validated with Zod schemas
- Schemas live in `lib/schemas/`
- Database types auto-generated from Supabase
- Use TypeScript strict mode - **NO `any` types allowed**
- Prefer type inference over explicit typing when possible

Example:
```typescript
import { z } from 'zod'

const CabinetSchema = z.object({
  name: z.string().min(1, 'Name is required'),
  location: z.string().min(1, 'Location is required'),
  capacity: z.number().positive()
})

type CabinetInput = z.infer<typeof CabinetSchema>
```

#### Styling & UI Components

- **Tailwind CSS** for all styling - avoid inline styles or CSS modules
- **Shadcn/ui** components in `components/ui/` - DO NOT modify these directly
- Use `cn()` utility from `lib/utils` for conditional classes
- Follow mobile-first responsive design
- Use Lucide React for icons

**Adding Shadcn components**:
```bash
npx shadcn@latest add <component-name>
```

#### Realtime Features

- Supabase Realtime for live updates
- Subscriptions managed in `lib/realtime/`
- Use in Client Components only
- Clean up subscriptions on unmount

Example:
```typescript
'use client'
import { useEffect, useState } from 'react'
import { createClient } from '@/lib/supabase/client'

export function RealtimeCabinets() {
  const [cabinets, setCabinets] = useState([])
  const supabase = createClient()
  
  useEffect(() => {
    const channel = supabase
      .channel('cabinets-changes')
      .on('postgres_changes', {
        event: '*',
        schema: 'public',
        table: 'cabinets'
      }, () => {
        // Refetch data
      })
      .subscribe()
    
    return () => { supabase.removeChannel(channel) }
  }, [])
}
```

- ❌ Hardcode API endpoints or secrets
